# Firebase Cloud Messaging (FCM)

Because your architecture splits business logic (MySQL) from identity (Firebase), routing a notification from a sensor event to a user’s phone requires a specific coordination between your backend, your database, and FCM.

## 1. The Core Concept: How Token Mapping Works

You cannot send a notification directly to a "User ID" or a "Device ID." Instead, you send notifications to an FCM Registration Token. This is a unique string generated by the Firebase SDK on the user's mobile device (e.g., their Flutter app) that identifies that specific app installation.

To make this work, you must map these tokens in your MySQL database.

### 1.1 Update Your MySQL Schema

Add a table to track user device tokens. A user might log in on both a phone and a tablet, so map this as a one-to-many relationship:
``` SQL
CREATE TABLE user_device_tokens (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL, -- References your MySQL users table
    fcm_token VARCHAR(255) NOT NULL UNIQUE,
    device_type VARCHAR(50), -- 'ios', 'android'
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

## 2. Step-by-Step Notification Data Flow

Here is exactly how an IoT event turns into a ping on a user's phone:
#### Step 1: Token Registration (Mobile Side)
1. The user opens your app and logs in.
2. Your app requests push notification permissions from the OS.
3. The Firebase SDK generates an FCM token: bk3rn156p9m...
4. The app calls your backend API (passing the Firebase JWT for authentication) and sends this token.
5. Your backend saves or updates this token in the user_device_tokens table, tied to that user's MySQL user_id.

#### Step 2: The Trigger (Backend Side)
Let’s say a sensor reports a critical alert (e.g., tank level too high, or a security gate left open).
1. Your ingestion engine (Node-RED, Go API, or Python script) processes the incoming telemetry packet.
2. It evaluates the data against a threshold and realizes an alert needs to be fired for device_123.

#### Step 3: Resolving the Target
Your backend looks up who needs to receive the alert:
1. It queries MySQL to find which user_id owns device_123.
2. It then queries the user_device_tokens table to get all active fcm_token strings associated with that user_id.

#### Step 4: Dispatching via Firebase Admin SDK
Your backend uses the Firebase Admin SDK to post a payload to the FCM API. Here is a quick example of what that payload looks like conceptually in Node.js:
``` JavaScript
const admin = require('firebase-admin');

// Array of tokens fetched from your MySQL database
const registrationTokens = ['bk3rn156p9m...', 'another_token_abc...'];

function sendSensorAlert(tokens, alertMessage, targetPlatform) {
  // 1. Build the base message that applies to BOTH platforms
const message = {
  notification: {
    title: '⚠️ Critical Sensor Alert',
    body: 'Tank 3 has exceeded safe capacity thresholds!'
  },
  data: {
    deviceId: 'device_123',
    alertType: 'overflow',
    click_action: 'FLUTTER_NOTIFICATION_CLICK' // Directs the mobile app behavior
  },
  tokens: registrationTokens,
};

// 2. Add iOS-Specific Tuning if needed
  if (targetPlatform === 'ios') {
    message.apns = {
      payload: {
        aps: {
          sound: 'critical_alert.caf', // Custom iOS alert sound
          badge: 1,                    // Increments app icon badge
          'content-available': 1       // Wakes up the app in the background
        }
      }
    };
  }

  // 3. Add Android-Specific Tuning if needed
  if (targetPlatform === 'android') {
    message.android = {
      priority: 'high',
      notification: {
        sound: 'critical_alert_android', // Custom Android sound
        channelId: 'critical_alerts'     // Required for Android 8.0+ notification channels
      }
    };
  }

// Send the message
admin.messaging().sendEachForMulticast(message)
  .then((response) => {
    console.log(`${response.successCount} messages were sent successfully`);
    
    // Clean up expired tokens if Firebase tells you they are no longer valid
    if (response.failureCount > 0) {
      // Loop through responses and delete bad tokens from MySQL
    }
  });
}
  ```

  ### 3. Critical Gotchas for IoT Push Notifications

### 3.1. Token Housekeeping (Stale Tokens)
FCM tokens expire or become invalid when users uninstall the app or clear their data. When you send a multicast message using the Admin SDK, Firebase returns an array of results. If a token returns an error like messaging/registration-token-not-registered, your backend must immediately delete that token from your MySQL table. If you don't, your notification dispatches will slow down over time.

### 3.2. Notification Fatigue vs. Aggregation
IoT devices can spam data. If a temperature sensor stays at a critical $45^\circ\text{C}$ and reports every 10 seconds, you do not want to send 360 push notifications an hour to the user's phone.
* **The Solution:** Implement a "cooldown" or "debounce" lock in your backend logic. Before sending a notification, check a cache (like Redis) or a quick flag in MySQL to see if an alert was already sent for this device in the last 15 or 30 minutes.

### 3.3. Background Data Payloads
Firebase allows two types of keys in its payload: notification (which the OS forces onto the screen as a visual banner) and data (invisible key-value pairs).
* For IoT apps, always pass critical context inside the data block (like deviceId or sensorType). This allows your mobile app to parse the notification when tapped and navigate the user straight to the specific chart or asset page instead of just opening the homepage.
